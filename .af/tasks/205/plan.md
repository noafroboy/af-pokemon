# Task 205 — Audio System: Chiptune Music + SFX + Pokémon Cries

## Assumption Audit

### Ambiguities in the task description

1. **"all 21 species"** — The game's `src/game/data/pokemon.ts` contains exactly 21 species: IDs 1–20 (Bulbasaur line through Raticate) and 25 (Pikachu). The task description cites Geodude and Onix as examples but neither exists in the current game data. **Assumption:** CRY_PARAMS covers the 21 species actually present: 1-20 and 25.

2. **CryParams naming conflict** — `src/game/types/PokemonTypes.ts` already exports a `CryParams` interface with `{ pitch, length, volume }`. The task requires a richer audio `CryParams` with `{ baseFreq, modFreq, modDepth, duration, waveType, envelope }`. **Assumption:** The new type is defined in `src/game/audio/cry-params.ts` as the primary `CryParams`; files that import both use type aliases (`import type { CryParams as AudioCryParams }`). The existing `PokemonTypes.CryParams` is not modified (to avoid breaking existing code).

3. **SFX sprite sheet vs. individual files** — The task mentions both a `sfx-sheet.wav` sprite sheet approach AND individual file references. **Assumption:** Use individual file references (simpler, no sprite alignment needed, Howler supports per-file loading). `sfx-sprites.ts` maps IDs to individual `/assets/sfx/*.wav` paths.

4. **`node-web-audio-api` vs. manual PCM** — The task says "Uses 'node-web-audio-api' or offline AudioContext". `node-web-audio-api` is not in package.json. **Assumption:** Use pure Node.js Buffer-based PCM synthesis and WAV header encoding — zero new npm dependencies required.

5. **Map music track IDs** — Maps define `music` as strings like `"pallettown"`, `"route1"`, `"viridian"`, `"pokecenter"`, `"pewterGym"`. The task track IDs use kebab-case (`pallet-town`, `route-1`). **Assumption:** AudioManager maps both forms: `{ "pallettown": "pallet-town", "route1": "route-1", ... }`.

6. **POKEMON_CENTER music** — The task says "Pokemon Center heal → playMusic('pokemon-center')". Since `pokemon-center.json` has `"music": "pokecenter"`, the normalization map handles this.

7. **Gym/Viridian music** — `viridian-city.json` uses `"music": "viridian"`. No "gym" track is in the map data explicitly for viridian. `pewter-gym.json` uses `"music": "pewterGym"`. **Assumption:** `pewterGym` maps to `"gym-battle"` track.

8. **"text-blip throttled to max 1/100ms"** — Interpreted as: module-level `lastBlipTime` variable; if `Date.now() - lastBlipTime < 100`, skip the playSFX call.

9. **"playCry(speciesId) when Pokemon appears"** — This happens at battle start (INTRO phase in BattleRenderer). **Assumption:** Called in GameEngine when transitioning into BATTLE phase, after setting up the battle state.

### What the task does NOT specify (and our defaults)

- **Music crossfade implementation**: Tone.js doesn't natively crossfade between Transport states. We'll implement crossfade by linearly ramping the volume of the outgoing track from current → 0 over 500ms before calling `Tone.Transport.stop()` and starting the new track.
- **Volume defaults**: Music 70%, SFX 80%
- **Audio settings in Options menu**: Not wiring the MenuSystem OPTIONS screen to actual volume sliders (that's not in scope). Volume API exists, UI hookup is a separate task.
- **Trainer battle cry**: Task doesn't specify playing enemy trainer's Pokemon cry. We play the wild/enemy Pokemon cry only for wild encounters and starter selection.

### Risks & Open Questions

1. Can Tone.js v14 `Tone.Part` handle note arrays with null pitches (rests)? → Yes, null pitch = schedule silence (no note triggered).
2. Does jsdom (test environment) have a stub for `AudioContext`? → jsdom provides a partial stub; we'll mock it in tests.
3. Will Next.js SSR throw on `import * as Tone from 'tone'`? → Yes, Tone.js accesses `window` at import time. Solution: dynamic import inside client-only `useEffect` or lazy init in AudioManager behind `typeof window !== 'undefined'`.

## Approach Alternatives

### APPROACH A: Conservative

- Only implement AudioManager stub + SFX files + minimal Tone.js music (single track, no crossfade)
- Skip FM modulation in cry synthesis (use simple oscillator instead)
- Minimal GameEngine integration (music on map load only, no per-event SFX)
- **Effort:** M | **Risk:** Low
- **Trade-off:** Deliverable quickly with low breakage risk, but misses 60% of the spec requirements (cries, battle SFX, crossfade, full track library)

### APPROACH B: Ideal (Selected)

- Full spec implementation: all 9 tracks, FM synthesis for all 21 cries, 14 SFX WAV files generated by script, full GameEngine + BattleSystem + MenuSystem + DialogSystem integration
- Conservative risk mitigations applied: SSR guards everywhere, all methods wrapped in try/catch, no new npm deps, file splitting to respect 150-line limit
- **Effort:** L | **Risk:** Med
- **Trade-off:** More lines of code and integration surface, but complete audio experience as specified

## Approach Decision

**Chose Approach B** with Approach A's risk mitigations.

Rationale: The task is explicit and complete — it defines every file, every method, every SFX file, and all integration points. Delivering a partial implementation would leave the audio system non-functional. The risk is mitigated by:
- `try/catch` on every AudioManager method (audio never crashes the game)
- `typeof window` guards (SSR safety)
- No new runtime dependencies (uses pure Node.js for WAV generation, installs nothing)
- File splitting (no file exceeds 150 lines)

## Production-Readiness Checklist

### 1. Persistence
Volume settings (music volume, SFX volume) are stored in `localStorage` under keys `audioMusicVolume` and `audioSFXVolume`. AudioManager reads them at initialization and writes on `setMusicVolume()`/`setSFXVolume()`. Default values: music=0.7, sfx=0.8. No other audio state needs to survive page refresh (music restarts, which is expected browser behavior). **Iron Law satisfied** — no in-memory-only primary store.

### 2. Error Handling
Every public AudioManager method is wrapped in `try/catch` with `console.warn('[AudioManager]', err)`. The game loop in `GameEngine` already has its own try/catch. Howler.js has built-in error callbacks that log to console. Tone.js transport errors are caught. Missing SFX files produce a Howler warning, not a crash. `synthesizeCry` returns `null` on error (callers check for null). **All external calls handled.**

### 3. Input Validation
`playMusic(trackId)`: validates trackId exists in TRACK_DEFINITIONS before proceeding; warns and no-ops if not found.
`playSFX(sfxId)`: validates sfxId exists in SFX_FILES map.
`playCry(speciesId)`: validates speciesId exists in CRY_PARAMS; warns and no-ops otherwise.
`setMusicVolume(v)` / `setSFXVolume(v)`: clamps v to [0, 1].
`generate-sfx.mjs`: validates output directory exists before writing.
**N/A for server-side validation** — audio is entirely client-side.

### 4. Loading States
Audio loading is transparent to the user by design (chiptune music synthesizes in real-time via Tone.js — no file download). SFX WAV files are pre-generated and served as static assets — Howler.js loads them lazily on first play, which may cause a brief delay on first SFX in a session. This is acceptable for a game of this scope. A "loading" state for audio specifically is N/A — the game canvas is the loading indicator already.

### 5. Empty States
If `audioContextUnlocked = false` (before user gesture), all audio operations silently no-op. The game is fully playable without sound — this is the empty state. No UI element signals "audio disabled"; the game just runs quietly until the first keypress.

### 6. Security
No API keys involved. No user-facing input to sanitize. `generate-sfx.mjs` reads only hardcoded parameter values (no user input, no file path injection). Tone.js and Howler.js are loaded from `node_modules` (package-lock.json validated). **N/A for API keys.**

### 7. Component Size
All new files planned under 150 lines:
- `cry-params.ts`: ~140 lines (21 species × 3 lines each in compact notation + interface + synthesizeCry)
- `tracks/overworld-tracks.ts`: ~130 lines
- `tracks/battle-tracks.ts`: ~130 lines
- `chiptune-tracks.ts` (barrel): ~25 lines
- `sfx-sprites.ts`: ~50 lines
- `AudioManager.ts`: ~145 lines
- `scripts/sfx-synth.mjs`: ~150 lines
- `scripts/generate-sfx.mjs`: ~80 lines
- Test files: ~60-100 lines each

Modified files (`GameEngine.ts` at 418 lines, `BattleSystem.ts` at 111 lines) already exist over 150 lines in GameEngine's case — we add minimal lines only.

### AudioManager Internal Flow

```
First user gesture
  └─ InputManager.onFirstGesture() → AudioManager.onFirstGesture()
       ├─ Tone.start() [resume AudioContext]
       ├─ (Howler resumes automatically on first play)
       └─ audioContextUnlocked = true

playMusic(trackId)
  ├─ Validate trackId in TRACK_DEFINITIONS
  ├─ If currentTrack === trackId, return early
  ├─ Crossfade: ramp currentGainNode to 0 over 500ms
  ├─ Build new Tone.Sequence from track notes
  ├─ Set BPM via Tone.Transport.bpm.value
  ├─ Start new sequence + Tone.Transport.start()
  └─ currentTrack = trackId

playSFX(sfxId)
  ├─ Guard: !audioContextUnlocked || SFX disabled → return
  ├─ Validate sfxId in SFX_FILES
  ├─ howlerInstance.play(sfxId)  [or new Howl({src}).play()]
  └─ Catch all errors silently

playCry(speciesId)
  ├─ Guard: !audioContextUnlocked → return
  ├─ Validate CRY_PARAMS[speciesId] exists
  ├─ const buf = synthesizeCry(audioCtx, params)
  ├─ const source = audioCtx.createBufferSource()
  ├─ source.buffer = buf; source.connect(audioCtx.destination)
  └─ source.start()
```

## Notes on Tone.js v14 API

- `Tone.Transport` → controls global BPM and play/stop
- `Tone.Sequence` → schedules repeating note arrays
- `Tone.Synth` with `oscillator.type: 'square'` for chiptune sound
- `Tone.Destination` for volume control
- Import: `import * as Tone from 'tone'` — but MUST be dynamic (browser-only)
- Pattern: `const Tone = await import('tone')` inside `onFirstGesture()`

## Notes on Howler.js v2 API

- `new Howl({ src: ['/assets/sfx/menu-select.wav'], volume: 0.8 })` for individual files
- `.play()` to trigger
- `Howler.volume(0.8)` to set global SFX volume
- Loaded lazily on first access — no preloading needed
